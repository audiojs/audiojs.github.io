## Design Context

### Users
Mixed technical audience: JavaScript audio developers, DSP-minded engineers, and advanced audio users evaluating package quality, standards coverage, and implementation seriousness.

Primary job to be done: establish authority and technical depth quickly, so the package feels credible, well-structured, and ready for production use.

### Brand Personality
Institutional, exacting, and acoustically literate.

Emotional goals: confidence, trust, rigor, and a sense that the library belongs equally in standards work, research notes, and studio tooling.

### Aesthetic Direction
Light mode with dark accents.

Primary references: secret research institute documents, W3C specifications, and pro audio manuals.

Supporting reference: restrained Bauhaus structure and typography, used for hierarchy and geometry rather than poster-like exuberance.

Avoid: generic startup gradients, soft SaaS cards, playful consumer UI, glossy futurism, and decorative dark-mode glow.

### Design Principles
1. Lead with authority: documentation should read like a technical file, not marketing.
2. Use geometry with restraint: grids, rules, stamps, and labels should feel precise and purposeful.
3. Keep audio visible: signal flow, response curves, and waveform evidence should anchor the design.
4. Favor contrast over ornament: light paper field, dark instrumentation, and one disciplined accent color.
5. Make structure legible at a glance: metadata, standards, architecture, and install path should each have a clear place.
